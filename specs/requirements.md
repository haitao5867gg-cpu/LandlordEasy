# 房屋收租系统(LandlordEasy)需求文档

> 状态:v1 已与房东确认 | 更新:2026-07-20
> 本文档是开发的需求依据,Kiro 开发前必读。技术方案见 design.md。

## 1. 项目概述

面向自有公寓运营的收租管理系统。业务背景:承包地皮自建公寓,共 4~5 栋楼、约 300 间房,全部整套出租(一个房间号对应一份合同),由 3 位房东(家人)共同管理。

- 房东端:手机 Web(H5)
- 租客端:微信公众号(服务号)内 H5
- 电脑端:后置(nice to have)
- 核心目标:跑通「出租 → 账单 → 提醒 → 收款 → 对账」闭环,300 间房量级下一切以自动化、批量化优先

## 2. 用户与登录

### 2.1 房东(3 人)
- 共同管理所有房产,权限相同,数据完全共享
- 登录:微信网页授权 + 后台白名单(openid 在名单内才放行),无需密码

### 2.2 租客
- 关注公众号 → 打开 H5 → 微信网页授权静默获取 openid
- 首次访问用**邀请码**绑定租约(房东签约时生成邀请码发给租客),一次绑定终身免登录
- 不做账号密码体系

## 3. 房屋管理

### 3.1 楼栋管理
- 维护 4~5 栋楼(名称/编号)
- 浏览层级:楼栋 → 房间号

### 3.2 房型模板(重要)
- 定义若干标准房型(如:标准单间、大单间、一室一厅)
- 模板字段:规格描述、配置清单(家具家电)、默认挂牌租金、默认押金规则、默认付款周期
- 房间关联房型后自动带出信息,个别房间可覆盖修改
- 目的:300 间房快速录入

### 3.3 房间档案
- 必填:楼栋、房间号、楼层(房间号含楼层信息,但楼层单独作为字段)、房型
- 可后补:照片、配置清单覆盖、备注
- 不需要:小区、单元、产权归属

### 3.4 状态管理
- 状态:空置 / 已租 / 维修中,随租约操作自动流转(退租 → 自动空置)
- 空置看板:空置房间列表、空置天数、按楼栋分组(空置率为核心经营指标)

### 3.5 到期预警
- 租约剩余 30/15/7 天到期的房间列表,辅助续租/招租决策

### 3.6 交接管理
- 入住交接:物品清单确认
- 退房核对:对照清单记录损坏与扣款依据
- 水电底数不需要记录(见 3.8)

### 3.7 维修记录
- 每间房的维修历史与费用,第一版房东手动录入
- 租客端在线报修为后续功能

### 3.8 水电(明确不做)
- 楼内为联网智慧水电表,租客自行在第三方平台缴费
- 第一版完全不涉及;数据模型预留扩展位,未来可接智慧水电 API

### 3.9 房间详情与历史档案(重要)

房间是长期资产,租客是流动的。单个房间的详情页必须聚合该房间的完整历史,数据只归档、永不删除:

- 当前状态:在租租客、当前租约、当前欠费情况
- 历史租约列表:每任租客、租期、租金、退租日期与退租原因
- 历史账单与收款记录:该房间历来所有账单、每笔支付(渠道/时间/金额)
- 押金记录:每任租约的押金收退与扣款
- 历史合同附件(合同电子化后)
- 维修记录
- 操作日志:该房间的关键操作留痕(状态变更、租约操作、收款确认等,记录操作人与时间)

退租不是删除数据,只是当前租约终结归档;换了十任租客,十任的记录都能在房间详情里追溯。

## 4. 租约管理

- 新签:录入租客(姓名、手机号,身份证可选)、租期、月租金、押金、付款周期、固定附加费用项(如清洁费)→ 自动生成邀请码
- 续签:延续老租约,租金可调整
- 退租:选定退租日 → 押金结算(退还/扣款,记录原因)→ 房间自动变空置
- 转租/换房:第一版由房东手动操作(退租+新签),租客端在线申请为后续功能
- 收租日:**按各自合同起租日独立滚动计算**,无统一收租日
- 押金:按租约单独设置,无全局规则

## 5. 账单与收款

### 5.1 账单自动生成(系统核心)
- 按租约账期自动生成每期账单,无需人工开单
- 账单构成:租金 + 租约约定的固定附加费用项(如清洁费),单期账单可手动追加临时费用项

### 5.2 滞纳金
- 账单逾期后可追加「滞纳金」费用项
- 默认金额 = 当期租金,可人工调整
- 是否追加由房东决定(不强制自动)

### 5.3 支付渠道(两阶段)
- **阶段一(先行开发)**:收款码支付。租客在 H5 查看账单 → 识别收款码付款 → 点「我已付款」(可选上传截图)→ 房东端「待确认」列表一键确认到账
- **阶段二(资质就绪后开启)**:微信支付商户号 JSAPI 支付。租客 H5 内直接支付,支付回调自动销账,无需人工确认
- 始终支持:房东手动记账(现金 / 银行转账)
- 支付独立建模:每笔支付记录含渠道、金额、时间、状态、凭证,与账单关联;架构上两阶段共用同一模型,阶段二仅是新增渠道

### 5.4 押金台账
- 押金收取、退还、扣款全程留痕

## 6. 催租提醒

- 通道:服务号模板消息
- 默认规则(参数可配置):到期前 3 天提醒一次 → 到期日提醒一次 → 逾期后每 3 天提醒一次,直至结清
- 房东端逾期看板:欠租人、欠租天数、金额,按楼栋分组

## 7. 经营报表

- 月度应收 vs 实收、收缴率(支持按楼栋分组)
- 空置率、空置损失估算
- 押金总额

## 8. 租客端功能清单

**访问权限规则:租客端一切由租约状态驱动,房东无需手动管理权限**
- 在租中:正常查看账单、交租
- 已退租但有未结清账单:仍可进入,只展示并可支付待结清账单
- 已退租且结清:显示「当前无在租房源」,可只读查看自己的历史账单;不再产生新账单
- 同一微信用户可先后绑定多份租约(离开后再回来租,用新邀请码绑定即可,历史保留)

第一版:
- 邀请码绑定租约
- 查看自己的账单(待付 / 已付 / 逾期)
- 收款码支付 + 「我已付款」上报
- 接收催租模板消息

后续版本:
- 查看租房合同
- 转租 / 换租 / 提前退房在线申请
- 在线报修
- 微信支付直接付款(随阶段二开启)

## 9. 前置条件与非功能要求

- **资质**:需注册个体工商户,用于(1)服务号认证获得模板消息能力(2)申请微信支付商户号(阶段二)。开发不阻塞于资质,收款码方案可先行
- 规模:约 300 间房、300+ 租客,并发低,单体应用即可,无需分布式
- 所有高频操作必须支持批量;禁止「逐间点击」式交互设计
- **数据只归档不删除**:租约、账单、支付等业务数据不允许物理删除,保证房间历史可完整追溯
- **操作日志**:关键写操作(租约签退、收款确认、金额修改等)记录操作人和时间——3 位房东共同管理,必须能查到「谁改的」

## 10. 阶段规划

- **P1(MVP)**:楼栋/房型/房间管理、租约管理、账单自动生成、收款码收款闭环、催租提醒、逾期与空置看板
- **P2**:微信支付自动销账、合同查看、经营报表完善、交接管理
- **P3**:租客在线申请(转租/退房)、报修、电脑端、智慧水电 API 对接
